# Sales Payment Structure Reference

## Purpose
This document describes the sales payment implementation added to the project so it can be referenced in future requests.

The implementation follows the same architectural style already used by `Sales`, `Products`, and `Customers`:
- DDD-oriented layering
- domain rules inside the `Sale` aggregate
- application use cases with `Command`, `Handler`, `Validator`, `Result`, and `Profile`
- EF Core mapping and repository persistence
- thin Web API controllers

## Business Model

### PaymentType Enum
`PaymentType` is persisted as `int` and follows this contract:

```text
1 = Cash
2 = CreditCard
3 = DebitCard
4 = Pix
```

`0 = Unknown` exists only as an invalid/default guard and must not be used in valid flows.

## Sale Payment Rules

### Allowed statuses
Payments can be registered only when the sale is in:

```text
2 = Subtotalized
3 = PaymentCompleted
```

Payments must be rejected when the sale is in:
- `Open`
- `EmittingNfce`
- `PrintingFiscalReceipt`
- `IntegratedWithErp`
- `Canceled`

### Version-based concurrency
The payment endpoint requires the current sale `Version`.

Behavior:
- the request must send the current `Version`
- the application handler compares the incoming value with the persisted sale version
- if they differ, the payment is rejected with `Sale version is outdated.`
- every successful payment changes the sale and generates a new `Version`

This guarantees optimistic concurrency for multiple operators or terminals trying to pay the same sale at the same time.

### Payment accumulation
- one request registers exactly one payment record
- the sale can receive multiple payments in multiple payment types
- `PaymentAmountTotal` is always the accumulated sum of all payment records

### Change generation
- only `Cash` can generate change
- `CreditCard`, `DebitCard`, and `Pix` never generate change
- if a non-cash payment would make the accumulated paid amount exceed the sale total, the request is rejected
- `ChangeAmountTotal` stores the effective current change value for the sale
- `SalesChanges` stores the audit history of generated change

### Status transitions
- first accepted payment moves the sale from `Subtotalized` to `PaymentCompleted` when the sale is still not fully paid
- when accumulated paid amount is equal to the sale total, the sale moves to `EmittingNfce`
- when accumulated paid amount is greater than the sale total and the last payment is `Cash`, change is generated and the sale moves to `EmittingNfce`

### Aggregate behavior
The `Sale` aggregate now exposes:

```text
RegisterPayment(PaymentType typePayment, decimal value)
```

Responsibilities:
- validate payment status eligibility
- validate payment type and value
- append payment history
- calculate accumulated payment amount
- calculate change only for cash
- reject invalid overpayment by non-cash methods
- update sale status
- renew the sale `Version`

### Application flow summary
1. Validate `saleId`, `version`, `typePayment`, and `value`
2. Load the sale from repository
3. Validate optimistic concurrency through `Version`
4. Validate allowed sale status
5. Register payment in the aggregate
6. Persist changes
7. Return the complete updated sale, including:
   - items
   - payments
   - changes
   - totals
   - status
   - new `Version`

## Current Scope Boundaries

### Included
- sale payment registration
- optimistic concurrency using sale `Version`
- multiple payment methods
- automatic change generation for cash
- payment/change history in read endpoints

### Not Included
- payment cancellation/reversal
- dedicated payment listing endpoint
- payment authorization integrations
- external acquirer or PIX provider communication
- automatic fiscal emission after payment
